Lab: Troubleshoot Edge Devices Using Toolbx Containers

Estimated reading time: 11 minutes.

Objective

Troubleshoot issues with edge devices by running toolbox containers.

Pending review

Before you Begin

You need a development machine running RHEL, to which you have unrestricted sudo access. The development machine has access to mirrors of RHEL package repositories and bootc container images.

These instructions were tested on RHEL 10.0 but should work with minimal or no changes on newer and older RHEL releases, since 9.6.

If you are using the course classroom, log in on the workstation VM as the user student with password student. If not, please adapt the instructions to your test environment.

Make sure you verified your test environment by following the first lab of this course.

You do not need any of the outcomes from the previous three activities, and could skip them. But you must execute the preparation tasks from the first lab of this course.

Instructions

You will experiment with another approach to add troubleshooting tools which might be missing in an image mode system: creating toolbx containers, which continue available after reboot, until you explicitly delete them.

  1. Prepare to start this activity by checking that you have the outcomes from previous activities on your development machine.

    1. Verify that you have a clone of the sample code repository of this course in your home directory.

      $ cd
$ ls rhde-bootc-samples/
LICENSE     bib-cloud-init  bib-qcow2 ...

    2. If needed, login to your private registry.

      $ podman login -u student -p redhat registry.lab.example.com:5000
Login Succeeded!

    3. Verify that you have a bootc container image named httpd-system:v1.0 in your private container registry. If you do not, please perform the first lab, which builds and publishes that image.

      $ skopeo inspect --format '{{ json .RepoTags }}' docker://registry.lab.example.com:5000/httpd-system:v1.0
["v1.0"]

    4. Verify that you have access to the local VM which is your test machine by using libvirt’s session interface. That VM will be your test machine.

      If you do not, please perform the preparation tasks from the first lab.

      $ virsh list --all
 Id   Name               State
-----------------------------------
 -    bootc-iso          shut off
      It is fine if you have other local VMs on your system.

    5. If your bootc-iso VM is not running (like in the output of the previous command), start it.

      $ virsh start bootc-iso
Domain 'bootc-iso' started

  2. Access your test machine and verify that it lacks a couple troubleshooting utilities.

    1. Enter the temp-test-iso directory and access your local VM using the SSH key in that directory.

      $ cd temp-test-iso
$ ssh -i edge-key -p 8022 core@127.0.0.1
Warning: Permanently added '[127.0.0.1]:8022' (ED25519) to the list of known hosts.
...
[core@bootc-iso ~]$

    2. Find which is the bootc container image that is currently booted.

      The password for the core user is redhat123.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:{httpd-system-v1-hash-aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa}
       Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

    3. Attempt to retrieve information about the system’s NUMA architecture, using utilities from the hardware locality package, and find that they are not installed.

      $ lstopo
-bash: lstopo: command not found
$ rpm -q hwloc
package hwloc is not installed

    4. Also attempt to get information about the system’s IO performance, and find that the Sysstat package is not installed either.

      $ iostat
-bash: iostat: command not found
$ rpm -q sysstat
package sysstat is not installed

    5. Leave the current running the SSH session to your test machine and open another terminal on your development machine to proceed to the next step.

  3. Make a toolbx container available in the private registry.

    1. On your development machine, verify if there is a copy of standard UBI toolbox container on the private registry.

      $ podman search registry.lab.example.com:5000/toolbox
NAME                                         DESCRIPTION
registry.lab.example.com:5000/ubi10/toolbox
$ skopeo inspect docker://registry.lab.example.com:5000/ubi10/toolbox:10.0
{
    "Name": "registry.lab.example.com:5000/ubi10/toolbox",
...

    2. If the copy is not available, your output will be similar to the following.

      $ podman search registry.lab.example.com:5000/toolbox
$ skopeo inspect docker://registry.lab.example.com:5000/ubi10/toolbox:10.0
FATA[0000] Error parsing image name "docker://registry.lab.example.com:5000/ubi10/toolbox:10.0": reading manifest 10.0 in registry.lab.example.com:5000/ubi10/toolbox: manifest unknown
      Remember that the podman search command and its equivalents with other container engines are NOT a reliable way to determine if a container image exists in an OCI container registry, because of limitations inherited from the Docker Distribution API.

    3. If the copy is not available, create it so you can keep with the scenario of disconnected deployments.

      $ skopeo copy --remove-signatures docker://registry.access.redhat.com/ubi10/toolbox:10.0 docker://registry.lab.example.com:5000/ubi10/toolbox:10.0
..
Writing manifest to image destination

  4. Use a toolbx container to collect information about the NUMA architecture of your test machine.

    1. Switch to the terminal on your _test machine) and use the copy of the toolbx image to create a toolbx container.

      $ podman login -u student -p redhat registry.lab.example.com:5000
Login Succeeded!
$ toolbox create --image registry.lab.example.com:5000/ubi10/toolbox:10.0
Image required to create Toolbx container.
Download registry.lab.example.com:5000/ubi10/toolbox:10.0 (245.1MB)? [y/N]: y
Created container: rhel-toolbox-10.0
Enter with: toolbox enter

    2. Enter the toolbox container you just created.

      Notice the change of the system prompt, which doesn’t display the current directory anymore, but includes an hexagonal bullet.

      $ toolbox enter
⬢ [core@toolbx ~]$ pwd
/var/home/core

    3. View the file systems which are backed by actual disk contents and which are overlays. Notice that the list of file systems of either time is much longer than outside of the toolbx container, if you compare with the outputs to the same commands from the previous lab.

      $ mount -t xfs
/dev/vda3 on /run/host/etc type xfs (rw,relatime,seclabel,attr2,inode64,logbufs=8,logbsize=32k,noquota)
/dev/vda3 on /run/host/sysroot type xfs (ro,relatime,seclabel,attr2,inode64,logbufs=8,logbsize=32k,noquota)
...
$ mount -t overlay
composefs on /run/host type overlay (ro,relatime,seclabel,lowerdir+=/run/ostree/.private/cfsroot-lower,datadir+=/sysroot/ostree/repo/objects,redirect_dir=on,metacopy=on)
...

    4. Because the toolbx container image does NOT contain the package repository settings for the classroom environment, copy them from the host machine. Also delete the configuration for the UBI repositories.

      $ sudo cp /run/host/etc/yum.repos.d/rhel_mirror.repo /etc/yum.repos.d/
$ sudo rm /etc/yum.repos.d/ubi.repo
      You could create a custom toolbx image, including such settings and also your preferred choice of troubleshooting tools.

    5. Install the hardware locality package.

      $ sudo dnf install hwloc
...
Complete!

    6. Finally, use the text mode hardware locality viewer.

      $ lstopo-no-graphics

  NUMANode L#0 (P#0 3909MB)
  Package L#0 + L3 L#0 (16MB) + L2 L#0 (4096KB) + L1d L#0 (32KB) + L1i L#0 (32KB) + Core L#0 + PU L#0 (P#0)
  Package L#1 + L3 L#1 (16MB) + L2 L#1 (4096KB) + L1d L#1 (32KB) + L1i L#1 (32KB) + Core L#1 + PU L#1 (P#1)
  HostBridge
    PCIBridge
      PCI 01:00.0 (Ethernet)
        Net "enp1s0"
    PCIBridge
      PCI 04:00.0 (SCSI)
        Block "vda"
    PCIBridge
      PCI 05:00.0 (Other)
    PCIBridge
      PCI 06:00.0 (Other)
    PCI 00:1f.2 (SATA)
      Block(Removable Media Device) "sr0"

    7. Exit the toolbx container and return to your test machine shell.

      $ exit
logout

  5. Verify that toolbx containers are application containers, and you can have multiple of them, each with their own container file system.

    1. Verify that toolbx containers are application containers, which you can manage using Podman.

      $ toolbox list
IMAGE ID      IMAGE NAME                                        CREATED
261e188dd142  registry.lab.example.com:5000/ubi10/toolbox:10.0  2 weeks ago

CONTAINER ID  CONTAINER NAME     CREATED      STATUS   IMAGE NAME
2251b66a79b8  rhel-toolbox-10.0  3 hours ago  running  registry.lab.example.com:5000/ubi10/toolbox:10.0
$ podman image list
REPOSITORY                                   TAG         IMAGE ID      CREATED      SIZE
registry.lab.example.com:5000/ubi10/toolbox  10.0        261e188dd142  2 weeks ago  1.18 GB
$ podman ps
CONTAINER ID  IMAGE                                             COMMAND               CREATED      STATUS         PORTS       NAMES
2251b66a79b8  registry.lab.example.com:5000/ubi10/toolbox:10.0  toolbox --log-lev...  3 hours ago  Up 10 minutes              rhel-toolbox-10.0

    2. Create another toolbx container, this time giving it a name.

      $ toolbox create stat --image registry.lab.example.com:5000/ubi10/toolbox:10.0
Created container: stat
Enter with: toolbox enter stat

    3. Verify that the new toolbx container is not running, because you didn’t enter yet.

      $ toolbox list
IMAGE ID      IMAGE NAME                                        CREATED
261e188dd142  registry.lab.example.com:5000/ubi10/toolbox:10.0  2 weeks ago

CONTAINER ID  CONTAINER NAME     CREATED        STATUS   IMAGE NAME
e6876646ee2e  rhel-toolbox-10.0  3 hours ago    running  registry.lab.example.com:5000/ubi10/toolbox:10.0
b8952ec51a35  stat               6 seconds ago  created  registry.lab.example.com:5000/ubi10/toolbox:10.0
$ podman ps -a
CONTAINER ID  IMAGE                                             COMMAND               CREATED         STATUS         PORTS       NAMES
e6876646ee2e  registry.lab.example.com:5000/ubi10/toolbox:10.0  toolbox --log-lev...  23 hours ago    Up 15 minutes              rhel-toolbox-10.0
b8952ec51a35  registry.lab.example.com:5000/ubi10/toolbox:10.0  toolbox --log-lev...  45 seconds ago  Created                    stat

    4. Enter the second toolbx container

      $ toolbox enter stat
⬢ [core@toolbx ~]$

    5. Configure the second toolbx container for your disconnected environment and install the Sysstat package.

      $ sudo cp /run/host/etc/yum.repos.d/rhel_mirror.repo /etc/yum.repos.d/
$ sudo rm /etc/yum.repos.d/ubi.repo
$ sudo dnf install sysstat
...
Complete!

    6. Get information about I/O performance of your test machine.

      $ iostat
Linux 6.12.0-55.24.1.el10_0.x86_64 (bootc-iso)  10/13/25        x86_64        (2 CPU)

avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           0.68    0.00    0.98    0.53    0.07   97.73

Device             tps    kB_read/s    kB_wrtn/s    kB_dscd/s    kB_read    kB_wrtn    kB_dscd
sr0               0.08         2.64         0.00         0.00       2104          0          0
vda              15.38       393.80        63.56         0.00     313408      50585          0

  6. Verify that toolbx containers are partially isolated from their host system (that is, your test machine) settings and application data.

    1. Create a file on your home directory.

      $ id
uid=1000(core) gid=10(wheel) groups=10(wheel)
$ echo '1,2,3 testing' > $HOME/test.txt

    2. Attempt to change the time zone of your toolbx container.

      $ timedatectl show -p Timezone
Timezone=America/Cancun
$ sudo timedatectl set-timezone 'Asia/Tokyo'
Failed to set time zone: Access denied

    3. Install and attempt to start a MySQL Server.

      $ sudo dnf install mysql8.4-server
...
Complete!
$ sudo systemctl start mysqld
Failed to connect to system scope bus via local transport: No such file or directory

    4. Verify that, despite the failure to start the database server, there are configuration files and data directories crated by installing its package.

      $ ls /etc/my.cnf.d
client.cnf  mysql-server.cnf
$ ls -a /var/lib/mysql
.  ..

    5. Exit the toolbx container and return to your test machine shell.

      $ exit
logout

  7. Verify if any of the changes made while in the second toolbox container affect your test machine.

    1. Verify if there is text file on your home directory.

      $ ls $HOME/txt
/var/home/core/test.txt
$ *cat $HOME/test.txt
1,2,3 testing

    2. Verify if there are MySQL Server configuration and data files.

      $ ls /etc/my.cnf.d
client.cnf  mysql-server.cnf
$ ls -a /var/lib/mysql
.  ..

  8. Verify that you can still enter your two toolbx containers after reboot.

    1. Reboot your test machine.

      $ sudo systemctl reboot --now
...
Connection to 127.0.0.1 closed.

    2. Give it a few moments to reboot and reconnect using SSH

      $ ssh -i edge-key -p 8022 core@127.0.0.1
...
[core@bootc-iso ~]$

    3. Verify that you still have two toolbx containers.

      $ toolbox list
IMAGE ID      IMAGE NAME                                       CREATED
261e188dd142  registry.lab.example.com:5000/ubi10/toolbox:10.0  2 weeks ago

CONTAINER ID  CONTAINER NAME     CREATED        STATUS  IMAGE NAME
2251b66a79b8  rhel-toolbox-10.0  3 hours ago    exited  registry.lab.example.com:5000/ubi10/toolbox:10.0
dab414a31a73  stat               5 minutes ago  exited  registry.lab.example.com:5000/ubi10/toolbox:10.0

    4. Enter the first toolbx container and verify it still contains the hardware locality utilities.

      $ toolbox enter
⬢ [core@toolbx ~]$ rpm -q hwloc
hwloc-2.11.1-3.el10.x86_64
⬢ [core@toolbx ~]$ exit
logout

    5. Enter the second toolbx container and verify it still contains the Sysstat package.

      $ toolbox enter stat
⬢ [core@toolbx ~]$ rpm -q sysstat
sysstat-12.7.6-2.el10.x86_64
⬢ [core@toolbx ~]$ exit
logout

    6. If you wish, stop and delete your toolbx containers using Podman.

      $ podman stop rhel-toolbox-10.0
rhel-toolbox-10.0
$ podman rm rhel-toolbox-10.0
rhel-toolbox-10.0
$ podman stop stat
stat
$ podman rm stat
stat
$ podman image rm registry.lab.example.com:5000/ubi10/toolbox:10.0
Untagged: registry.lab.example.com:5000/ubi10/toolbox:10.0
Deleted: 261e188dd1428d36b03a512e4f67a7285ed6f68a743e07efbdd16e70473e680d
$ toolbox list

You created two toolbx containers, installed different packages in each, and verified what they can and cannot affect their host system. Then you cleaned up your test machine to get rid of the toolbx containers.
