Lab: Roll Back Updates to Edge Devices

Estimated reading time: 7 minutes.

Objective

Revert edge devices to a previous known good bootc container image which is available in the device itself.

Pending review

Before you begin

You need a development machine running RHEL, to which you have unrestricted sudo access. The development machine has access to mirrors of RHEL package repositories and bootc container images.

These instructions were tested on RHEL 10.0 but should work with minimal or no changes on newer and older RHEL releases, since 9.6.

If you are using the course classroom, log in on the workstation VM as the user student with password student. If not, please adapt the instructions to your test environment.

This activity is a continuation of the previous lab of this course. You must perform the two activities in sequence, without destroying your classroom environment or its VMs.

Instructions

You will perform manual day-2 configurations to an image mode system, to emulate changes that would occur to real systems, and then roll back that system to a previously installed bootc container image. After rolling back, you will verify that system configurations were also rolled back to the state from before the update, but application data retains changes performed after the update.

  1. Prepare to start this activity by checking that you have the outcomes from previous activities on your development machine.

    1. Verify that you have a clone of the sample code repository of this course in your home directory.

      $ cd
$ ls rhde-bootc-samples
LICENSE  bib-cloud-init  bib-qcow2 ...
...

    2. Verify that you have access to the local VM named bootc-iso by using libvirt’s session interface. That VM is your test machine.

      If you do not have access, please perform the previous lab.

      $ virsh list --all
 Id   Name               State
-----------------------------------
 -    bootc-iso          shut off
      It is fine if you have other local VMs on your system.

    3. If your bootc-iso VM is not running (like in the output of the previous command), start it.

      $ virsh start bootc-iso
Domain 'bootc-iso' started

    4. Verify that you have a directory named temp-test-iso with files that were used to create and access your test machine.

      $ ls temp-test-iso/
bib-iso.sh  config.toml  edge-key  edge-key.pub  httpd-system.iso ...

  2. Access your test machine and perform day-2 configurations, which affect system settings and application data.

    1. Enter the temp-test-iso directory and access your local VM by using the SSH key in that directory.

      $ cd temp-test-iso
$ ssh -i edge-key -p 8022 core@127.0.0.1
Warning: Permanently added '[127.0.0.1]:8022' (ED25519) to the list of known hosts.
...
[core@bootc-iso ~]$

    2. Verify that there are two deployed bootc container images in the system, and that the currently booted image is the most recent one.

      The password for the core user is redhat123.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:f7b6046dc81cbb80e79bdfb58fcd1c17a3ab84ae47f3b21b1af384f2277d59dc
       Version: 10.0 (2025-09-30 22:54:29.224606481 UTC)

  Rollback image: registry.lab.example.com:5000/httpd-system:v1.0
          Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
         Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

    3. Verify that the system is configured with a non-UTC time zone and change it to a different non-UTC time zone.

      $ timedatectl show -p Timezone
Timezone=America/Cancun
$ sudo timedatectl set-timezone 'US/Alaska'
$ timedatectl show -p Timezone
Timezone=US/Alaska

    4. Change the default index page of the Apache HTTP Server, which contains files to mock a single-page web application. Also verify that the current system image is NOT serving the application from the default document root.

      $ cat /var/www/html/index.html
Hacked the webapp
$ sudo bash -c "echo 'Cleaned the webapp' > /var/www/html/index.html"
$ curl 127.0.0.1
Misc tests of bootc images and firewalld, v1.1

    5. Verify that Cockpit is installed and enabled.

      $ rpm -q cockpit
cockpit-334.1-1.el10_0.x86_64
$ sudo systemctl is-active cockpit.socket
active

  3. Reboot your test machine, so it reverts to its previous bootc container image.

    1. Verify that the bootc update timer is disabled.

      $ sudo systemctl is-active bootc-fetch-apply-updates.timer
inactive
      If the timer were enabled, it would be recommended to disable it before a roll back operation, to prevent another image update from overriding the roll back operation.

    2. Roll back the system.

      $ sudo bootc rollback
Next boot: rollback deployment

    3. Verify that the human-readable output of the bootc status command displays no indication that a roll back operation was queued.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:f7b6046dc81cbb80e79bdfb58fcd1c17a3ab84ae47f3b21b1af384f2277d59dc
       Version: 10.0 (2025-09-30 22:54:29.224606481 UTC)

  Rollback image: registry.lab.example.com:5000/httpd-system:v1.0
          Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
         Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

    4. Verify that the next boot uses the roll back image and that a roll back is queued for the next reboot, by using the bootc extended status.

      $ sudo bootc status --format json | jq -r '.spec.bootOrder'
rollback
$ sudo bootc status --format json | jq -r '.status.rollbackQueued'
true

    5. Reboot the system.

      $ sudo systemctl reboot --now
...
Connection to 127.0.0.1 closed.

  4. Access your test machine and verify how its configuration and application data were affected by rolling back its system image.

    1. Give your VM a few moments to reboot and reconnect to it by using SSH.

      $ ssh -i edge-key -p 8022 core@127.0.0.1
Warning: Permanently added '[127.0.0.1]:8022' (ED25519) to the list of known hosts.
...
[core@bootc-iso ~]$

    2. Verify that the booted image is no longer the most recent one.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
       Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

  Rollback image: registry.lab.example.com:5000/httpd-system:v1.0
          Digest: sha256:f7b6046dc81cbb80e79bdfb58fcd1c17a3ab84ae47f3b21b1af384f2277d59dc
         Version: 10.0 (2025-09-30 22:54:29.224606481 UTC)

    3. Verify that the currently booted image is the same for the next boot and that there is no roll back operation queued.

      $ sudo bootc status --format json | jq -r '.spec.bootOrder'
default
$ sudo bootc status --format json | jq -r '.status.rollbackQueued'
false

    4. Verify that the time zone changed to the value from before the update, that is, system configurations were also rolled back.

      $ timedatectl show -p Timezone
Timezone=America/Cancun

    5. Verify that the default web root directory contains data from after the update, that is, application data is NOT rolled back.

      $ cat /var/www/html/index.html
Cleaned the webapp

    6. Verify that, because the rolled back system image stores web application files on the default web root, it now serves the application as changed after the update, which may be unexpected.

      $ curl 127.0.0.1
Cleaned the webapp

    7. Verify that Cockpit is not available, because it was not present in the rolled back system image.

      $ rpm -q cockpit
package cockpit is not installed

    8. Verify that the auto-update time remains inactive, because it was masked before applying the update.

      $ sudo systemctl is-active bootc-fetch-apply-updates.timer
inactive

You observed that rolling back the system image also rolls back system settings (the /etc directory) because it is part of the bootc image deployment. You also observed that rolling back the system image does NOT roll back application data (the /var directory) because it is not part of the bootc deployment, and you may need to perform manual data cleaning and migration for the system to work after rolling back.
