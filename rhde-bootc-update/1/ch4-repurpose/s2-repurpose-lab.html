Lab: Switch System Images of Edge Devices

Estimated reading time: 12 minutes.

Objective

Repurpose edge devices by switching to a different bootc container image.

Before you begin

You need a development machine running RHEL, to which you have unrestricted sudo access. The development machine has access to mirrors of RHEL package repositories and bootc container images.

These instructions were tested on RHEL 10.0 but should work with minimal or no changes on newer and older RHEL releases, since 9.6.

If you are using the course classroom, log in on the workstation VM as the user student with password student. If not, please adapt the instructions to your test environment.

Make sure you verified your test environment by following the first lab of this course.

You do not need any of the outcomes from the previous two activities, and could skip them. But you must execute the preparation tasks from the first lab of this course.

Instructions

You will switch an image mode system to a different container image, but it will fail because of an RPM package which performs an action that is incompatible with image mode. You will troubleshoot the failure and find its root cause, then make a second attempt by using an updated bootc container image, which implements the recommended fix.

The classroom environment has an issue which requires setting SELinux to permissive mode. This is something Red Hat does NOT advise doing to any production system, including developer machines and CI/CD servers.

On your development machine, perform the following before you attempt any lab in this course.

$ sudo setenforce Permissive

Note that SELinux is still in enforcing mode on your server machine and also on all test VMs which stand in for managed devices. Only your development machine needs SELinux set to permissive mode.

  1. Prepare to start this activity by checking that you have the outcomes from previous activities on your development machine.

    1. Verify that you have a clone of the sample code repository of this course in your home directory.

      $ cd
$ ls rhde-bootc-samples/
LICENSE     bib-cloud-init  bib-qcow2 ...

    2. If needed, log in to your private registry.

      $ podman login -u student -p redhat registry.lab.example.com:5000
Login Succeeded!

    3. Verify that you have a bootc container image named httpd-system:v1.0 in your private container registry. If you do not, please perform the first lab, which builds and publishes that image.

      $ skopeo inspect --format '{{ json .RepoTags }}' docker://registry.lab.example.com:5000/httpd-system:v1.0
["v1.0"]

    4. Verify that you have access to the local VM which is your test machine by using libvirt’s session interface.

      If you do not have access, please perform the preparation tasks from the first lab.

      $ virsh list --all
 Id   Name               State
-----------------------------------
 -    bootc-iso          shut off
      It is fine if you have other local VMs on your system.

    5. If your bootc-iso VM is not running (like in the output of the previous command), start it.

      $ virsh start bootc-iso
Domain 'bootc-iso' started

  2. Access your test machine and verify it is running a system image which provides a mock of a single-page web application.

    1. Enter the temp-test-iso directory and access your local VM by using the SSH key in that directory.

      $ cd temp-test-iso
$ ssh -i edge-key -p 8022 core@127.0.0.1
Warning: Permanently added '[127.0.0.1]:8022' (ED25519) to the list of known hosts.
...
[core@bootc-iso ~]$

    2. Find the bootc container image that is currently booted.

      The password for the core user is redhat123.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
       Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

    3. Review the mock web application files.

      $ curl 127.0.0.1
Misc tests of bootc images and firewalld
$ cat /var/www/html/index.html
Misc tests of bootc images and firewalld
      If you did not skip the previous activities, and already performed the upgrade and roll back labs, you may have different contents on your index.html file.

    4. Leave the SSH session open because you will return to it shortly.

  3. Build and publish a bootc container image to replace the one in your test machine.

    The new image simulates a scenario of adding a server-side component to a browser-based application, by adding a PHP runtime to the system image.

    1. Open a new terminal and create a new temporary directory for building the new container image.

      $ cd
$ mkdir temp-php-bootc
$ cp -r rhde-bootc-samples/php-bootc/* temp-php-bootc
$ cd temp-php-bootc
$ ls
Containerfile  html  yum.repos.d
$ ls html/
app.php index.html

    2. Review the Containerfile for the new image, and notice the following:

      1. It adds the PHP runtime and its FastCGI engine, as the recommended way of deploying PHP on an Apache HTTP Server.

      2. It moves all web application files to a directory outside of /var so they cease to be application data but become part of the proper system image.

      FROM registry.lab.example.com:5000/rhel10/rhel-bootc:10.0
ADD yum.repos.d /etc/yum.repos.d
RUN dnf install -y firewalld cockpit httpd php-fpm && \
  dnf clean all && \
  firewall-offline-cmd --zone=public --add-service=cockpit && \
  firewall-offline-cmd --zone=public --add-service=http && \
  systemctl enable firewalld && \
  systemctl enable cockpit.socket && \
  systemctl enable httpd 
ADD html /webapp/html
RUN sed -i '1,$ s?/var/www/?/webapp/?' /etc/httpd/conf/httpd.conf && \
  chmod -R a+rX /webapp && \
  semanage fcontext -a -t httpd_sys_content_t "/webapp(/.*)?"

    3. Build and publish the bootc container image in the private registry.

      $ podman build -t php-bootc .
Successfully tagged localhost/php-bootc:latest
be3640228ea7872d099c6a7eaa6c994c3f0d48ff397f3ffbd32924b316ba58d4
$ skopeo copy containers-storage:localhost/php-bootc docker://registry.lab.example.com:5000/php-bootc:v1.0
...
Writing manifest to image destination

  4. Switch your test machine to the new bootc container image.

    1. Return to the terminal with an open SSH session to your test machine.

    2. Verify that the VM can access the private registry by checking for available updates.

      $ sudo bootc upgrade --check
No changes in: docker://registry.lab.example.com:5000/httpd-system:v1.0
      If you are continuing from the previous lab, which performed a roll back operation, the bootc update command will still report "no updates available" because it is smart enough to NOT retry an update that was already rolled back.

    3. If you skipped the previous two activities of this course, the bootc upgrade command will fail because your test machine does not have TLS certificates and credentials for accessing the private registry.

      In that case, configure your test machine for accessing the private registry.

      $ sudo bootc update --check
ERROR Upgrading: Creating importer: failed to invoke method OpenImage: failed to invoke method OpenImage: reading manifest v1.0 in registry.lab.example.com:5000/httpd-system: authentication required
$ scp student@utility:/opt/registry/certs/domain.crt
domain.crt
$ sudo cp domain.crt /etc/pki/ca-trust/source/anchors/
$ sudo update-ca-trust
$ sudo podman login -u student -p redhat registry.lab.example.com:5000
Login Succeeded!
$ sudo cp /run/containers/0/auth.json /etc/ostree/auth.json
$ sudo bootc upgrade --check
No changes in: docker://registry.lab.example.com:5000/httpd-system:v1.0

    4. Switch to the new bootc container image.

      $ sudo bootc switch registry.lab.example.com:5000/php-bootc:v1.0
...
Queued for next boot: registry.lab.example.com:5000/php-bootc:v1.0
  Version: 10.0
  Digest: sha256:cea5e54156044edc63d0f52c46890a90f0332e0d6616a8d3b8fc7b84afe1d002
      You could have used the --apply option, like in the previous labs, when you performed image upgrades.

    5. Verify that the new image is staged for the next boot.

      $ sudo bootc status
  Staged image: registry.lab.example.com:5000/php-bootc:v1.0
        Digest: sha256:cea5e54156044edc63d0f52c46890a90f0332e0d6616a8d3b8fc7b84afe1d002
       Version: 10.0 (2025-10-07 22:27:57.066802937 UTC)

● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
       Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

  Rollback image: registry.lab.example.com:5000/httpd-system:v1.0
          Digest: sha256:f7b6046dc81cbb80e79bdfb58fcd1c17a3ab84ae47f3b21b1af384f2277d59dc
         Version: 10.0 (2025-09-30 22:54:29.224606481 UTC)
      If you skipped the previous activity, you would not see a "rollback image" in the output of the bootc status command.

    6. Reboot your test machine.

      $ sudo systemctl reboot --now
...
Connection to 127.0.0.1 closed.

    7. Wait a few moments, to give your test machine time to reboot, and verify that it fails to run the updated web application.

      $ curl 127.0.0.1:8080
Single-page web app, now with a <href="app.php">server-side</a> component.
$ curl 127.0.0.1:8080/app.php
<!DOCTYPE HTML PUBLIC "-//IETF//DTD HTML 2.0//EN">
<html><head>

</head><body>
<h1>Service Unavailable</h1>

request due to maintenance downtime or capacity
problems. Please try again later.</p>
</body></html>

    8. Reconnect to your test machine so you can perform troubleshooting steps.

      Remember that your two terminals are in different working directories: one has the files to build the new bootc image, the other has the files to manage the local VM that is your test machine.

      $ basename $(pwd)
temp-test-iso
$ ssh -i edge-key -p 8022 core@127.0.0.1
...
[core@bootc-iso ~]$

  5. Troubleshoot the failure to run the new server-side component of your web application on your test machine.

    1. Verify that it is running the new bootc container image.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/php-bootc:v1.0
        Digest: sha256:cea5e54156044edc63d0f52c46890a90f0332e0d6616a8d3b8fc7b84afe1d002
       Version: 10.0 (2025-10-07 22:27:57.066802937 UTC)

  Rollback image: registry.lab.example.com:5000/httpd-system:v1.0
          Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
         Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

    2. Verify the status of the FastCGI module for PHP.

      You will see an error related to a missing directory inside the /var directory, which is application data.

      $ sudo systemctl status php-fpm
× php-fpm.service - The PHP FastCGI Process Manager
     Loaded: loaded (/usr/lib/systemd/system/php-fpm.service; disabled; preset: disabled)
     Active: failed (Result: exit-code) since Tue 2025-10-07 18:00:18 EST; 4min 11s ago
...
Oct 07 18:00:20 bootc-iso php-fpm[969]: [07-Oct-2025 18:00:18] ERROR: failed to open error_log (/var/log/php-fpm/error.log): No such file or directory (2)
...

    3. Confirm that the offending directory is really missing, instead of having either incorrect permissions or an incorrect SELinux context, on it or its parent directories, and that it is provided by the php-fpm package.

      $ sudo ls -ld /var/log/php-fpm
ls: cannot access '/var/log/php-fpm': No such file or directory
$ rpm -qf /var/log/php-fpm
php-fpm-8.3.15-1.el10.x86_64

  6. There could be additional files and directories that the package initializes which belongs to application data on image mode systems, but that didn’t appear in our simplistic test. Find these files, so you can deliver a proper fix for the bootc container image.

    Remember that changes to /var are NOT applied to an image mode system during image updates. This explains why the new bootc container image would work, if tested on a container engine, or if used to provision a new system, but fail on switch and update.

    1. Verify whether the package contains other files in /var.

      The additional files you see are required by PHP session management features.

      $ rpm -ql php-fpm | grep /var
/var/lib/php/opcache
/var/lib/php/session
/var/lib/php/wsdlcache
/var/log/php-fpm

    2. Verify that the package creates or changes files in /var from its RPM scriptlets.

      This package does not.

      $ rpm -q --scripts php-fpm
postinstall scriptlet (using /bin/sh):


if [ $1 -eq 1 ] && [ -x "/usr/lib/systemd/systemd-update-helper" ]; then
    # Initial installation
    /usr/lib/systemd/systemd-update-helper install-system-units php-fpm.service || :
fi
preuninstall scriptlet (using /bin/sh):


if [ $1 -eq 0 ] && [ -x "/usr/lib/systemd/systemd-update-helper" ]; then
    # Package removal, not upgrade
    /usr/lib/systemd/systemd-update-helper remove-system-units php-fpm.service || :
fi


# Raised by new pool installation or new extension installation

    3. Now that you have a diagnostic, roll back the test machine to its original image, so you can later retry switching to the new image, after you implement a fix.

      $ sudo bootc rollback --apply
Next boot: rollback deployment
$ sudo systemctl reboot --now
...
Connection to 127.0.0.1 closed.

  7. Update the new bootc container image to initialize application data at boot time instead of at build time. The fix uses Systemd to create all required directories with proper owner and permissions.

    1. Return to the terminal that you used to build the bootc image

      $ basename $(pwd)
temp-php-bootc

    2. Append the following line to the Containerfile.

      ADD tmpfiles.d /usr/lib/tmpfiles.d
      You can also replace the Containerfile with the one provided in the Containerfile.fix file.

    3. Review the configuration file for the Systemd tempfiles service in the tmpfiles.d/php-fpm.conf file.

      d /var/lib/php           0750 root   root   -
d /var/lib/php/opcache   0770 root   apache -
d /var/lib/php/session   0770 root   apache -
d /var/lib/php/wsdlcache 0770 root   apache -
d /var/log/php-fpm       0770 apache root   -

    4. Build and publish the fixed bootc container image in the private registry.

      $ podman build -t php-bootc .
Successfully tagged localhost/php-bootc:latest
be3640228ea7872d099c6a7eaa6c994c3f0d48ff397f3ffbd32924b316ba58d4
$ skopeo copy containers-storage:localhost/php-bootc docker://registry.lab.example.com:5000/php-bootc:v1.0
...
Writing manifest to image destination

  8. Return to the terminal used to access your test machine and try again switching to the new bootc container image, to verify that the fix works as intended.

    1. Reconnect to your local VM, which should be finished rebooting by now, from the rollback operation.

      $ ssh -i edge-key -p 8022 core@127.0.0.1
...
[core@bootc-iso ~]$

    2. Verify that it is running the original image.

      $ sudo bootc status
● Booted image: registry.lab.example.com:5000/httpd-system:v1.0
        Digest: sha256:75391a26d30f30bad03f0741de6c51dff10e7be26199c9b5ceed47019a61b511
       Version: 10.0 (2025-09-15 19:03:35.318301268 UTC)

  Rollback image: registry.lab.example.com:5000/php-bootc:v1.0
          Digest: sha256:9d89cffa42a74ed710bbd208677801089a63532d7370392493bea9e8b15a7e39
         Version: 10.0 (2025-10-07 23:28:56.653899779 UTC)

    3. Switch to the fixed bootc container image.

      $ sudo bootc switch registry.lab.example.com:5000/php-bootc:v1.0
...
Queued for next boot: registry.lab.example.com:5000/php-bootc:v1.0
  Version: 10.0
  Digest: sha256:2dcb16c2b73bc1d24eb1b90578cc91b95302c3b9e68c89f8a986b19564e24af9
$ sudo systemctl reboot --now
...
Connection to 127.0.0.1 closed.

    4. Give your test machine a few moments to reboot (again!) and verify that it now runs the updated system image successfully.

      $ curl 127.0.0.1:8080
Single-page web app, now with a <href="app.php">server-side</a> component.
$ curl 127.0.0.1:8080/app.php
PHP running at 172.25.250.9 under Apache/2.4.63 (Red Hat Enterprise Linux)

You experienced an issue that’s not uncommon when your repurpose an image mode system. It was caused by RPM packages that initialize application data at build time instead of at boot time. You diagnosed the issue, and then fixed it by using Systemd features.
