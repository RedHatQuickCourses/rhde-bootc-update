Rollback Updates to Image Mode Systems

Estimated reading time: 7 minutes.

Objective

Rollback updates to edge devices by using the bootc utility.

Work in progress

About transactional rolling back updates of image mode systems

As explained during the previous section, bootc updates are atomic and transactional regarding the state of the read-only system image, but not regarding the state of mutable sytem configurations and application data.

Consequently, roll backs may require aditional actions to be performed at runtime, by Systemd units embedded into a bootc image, or manually, as a day-2 activity, or even by day-2 automation. Else, the system configuration and application data may be left in a state that is not compatible with the system image it rolled back to.

Bootc rollback operation and system configurations

A bootc rollback operation is comparativelly simpler than bootc update or bootc switch operations. The rollback operation just changes the boot order of the existing bootc deployments, so the previous one boots instead of the current one.

The bootc rollback changes the system boot loader configuration to boot from the previous bootc container image deployment. That means it will boot from not just the Linux kernel from the previous bootc container image, but it also uses the root file system from the previous image, and it also uses the copy of /etc that was created at the time that image was originally deployed.

The rollback operation does NOT perform a new tree-way merge of the /etc directory, but uses whatever was the content of that directory in that deployment, which effectlively means system configurations are rolled back also. Local day-2 changes performed while that deployment were active are retained, but other day-2 changes performed after the update are lost.

Notice it is not a "reset" to the /etc contents from the previous bootc image: it is a "reset" to the mutable state that was in /etc right before the last update.

Application data during rollback.

After rollback, an image mode system remains using the same copy of the /var directory that was already in use before the operation, which was also the same as before the last update. Bootc rollbacks, like updates and switches do NOT change the /var directory in anyway.

So, if a bootc update resulted in changes to application data, for example migrating a relational database to a new schema, those changes may be incompatible with the system image booted after a rollback. It is the responsability of the bootc image itself, or if its administrator, to revert those changes, for example by restoring a database backup.
